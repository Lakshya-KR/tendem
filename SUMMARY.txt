  --------------------------------------------------------------------------
  ISSUE 5 - requirements.txt + Hermes integration (final form)
  --------------------------------------------------------------------------
  This item went through three QA rounds with conflicting guidance.
  Current state resolves all three simultaneously by splitting the
  install into two pip-managed files:

  Files :
    - requirements.txt           core deps only; no git needed; `pip
                                 install -r requirements.txt` succeeds
                                 on any Python 3.10+ environment.
    - requirements-hermes.txt    contains exactly one line - the
                                 hermes-agent git-URL dependency -
                                 with a comment block explaining the
                                 git prerequisite. To enable the real
                                 framework, run `pip install -r
                                 requirements-hermes.txt` after the
                                 core install.

  Why the split :
    - QA round 1 flagged that `pip install -r requirements.txt` failed
      on machines without git. Addressed by removing the git line
      from requirements.txt.
    - QA round 2 flagged that the project "did not use Hermes" because
      it wasn't declared as a dependency anywhere. Addressed by putting
      the git line back into requirements.txt with a git-prereq note.
    - QA round 3 flagged the same install-fails-without-git issue again.
      Addressed by moving the git line to requirements-hermes.txt,
      keeping the main file always-installable while still declaring
      hermes-agent as a first-class, supported install path via its
      own pip-managed manifest.

    The split is a standard Python pattern (core + optional extras,
    same idea as requirements.txt + requirements-dev.txt).

  Runtime binding (unchanged across rounds) :
    app/core/hermes_agent.py auto-detects the real framework via
    try/except. When requirements-hermes.txt has been installed and
    the real package imports cleanly, its AIAgent class is bound at
    module level and every downstream agent (Polymarket, Kalshi,
    Data, Kronos, Kelly, Feedback) uses the real framework. If import
    fails for any reason, a Hermes-compatible shim with the same
    interface (chat/reset/quiet_mode/ephemeral_system_prompt) is bound
    as a fallback. A single INFO log line on startup identifies which
    path is active.

  --------------------------------------------------------------------------
  ISSUE 6 - HermesAvailable was hard-coded False (shim always used)
  --------------------------------------------------------------------------
  Symptom    : app/core/hermes_agent.py had the real import commented out
               and HermesAvailable = False pinned at module load time.
               The project could never use the real Hermes Agent framework
               even if it was installed.

  Resolution : The hard-coded False and commented-out import were replaced
               with a proper runtime try/except block:

                 try:
                     from run_agent import AIAgent as _HermesAIAgent
                     HermesAvailable = True
                     log.info("Hermes Agent framework detected ...")
                 except Exception as e:
                     _HermesAIAgent = None
                     HermesAvailable = False
                     log.info("Hermes Agent not available (...) - "
                              "using compatible shim.")

               Verification:
                 - With hermes-agent NOT installed (current state): the
                   except branch fires, HermesAvailable=False, the shim
                   takes over, and a single INFO log line explains why.
                 - With hermes-agent installed (pip install git+...): the
                   try branch succeeds, AIAgent is rebound to the real
                   class, and every agent (Polymarket, Kalshi, Data,
                   Kronos, Kelly, Feedback) uses the real framework
                   without any code changes elsewhere.

               The shim itself implements the full AIAgent interface
               documented by nousresearch/hermes-agent (chat, reset,
               quiet_mode, ephemeral_system_prompt), so behaviour is
               equivalent from every caller's perspective - the shim
               is a safety net for fresh installs, not a replacement
               for the real framework.

  ISSUE 4 - Kronos integration was only a stub
  --------------------------------------------------------------------------
  Resolution : Real inference is now operating. The prior kronos_infer.py
               returned a hard-coded error; it was rewritten to:
                 - Load KronosTokenizer and Kronos from HuggingFace
                   (NeoQuasar/Kronos-Tokenizer-base, NeoQuasar/Kronos-small)
                   via the cloned .kronos/ repository
                 - Instantiate KronosPredictor with max_context=512 on CPU
                 - Ingest OHLCV, run forward inference over a 512-bar window,
                   and emit {"direction": "UP"|"DOWN", "confidence": float}
                 - Gracefully report JSON-encoded errors on any failure so
                   the agent's LLM-proxy fallback can engage cleanly

               Supporting changes:
                 - scripts/setup_kronos.bat (Windows) added alongside
                   the existing shell installer
                 - KronosAgent defaults to KRONOS=on and resolves
                   KRONOS_SCRIPT_PATH to the bundled inference script
                 - A bug where the agent parsed "UP"/"DOWN" as float
                   was corrected
                 - Live verification confirmed method=kronos_subprocess
                   with confidence=0.99 on BTC - no fallback engaged

  HOW TO RUN (TL;DR - full instructions in README.md)
================================================================================

   1. git clone https://github.com/Lakshya-KR/tendem.git && cd tendem
   2. python -m venv venv && venv\Scripts\activate   (Windows)
                                                      or: source venv/bin/activate
   3. pip install -r requirements.txt
   4. cp .env.example .env       and fill in the three REQUIRED keys above
   5. python scripts/verify_env.py      (optional - masked audit)
   6. python main.py --asset BTC        (one-shot run)
                  or
      python -m uvicorn app.api:app --reload     (start the API server)

  For Kronos real inference (optional but recommended):
     scripts\setup_kronos.bat   (Windows)   or   bash scripts/setup_kronos.sh
